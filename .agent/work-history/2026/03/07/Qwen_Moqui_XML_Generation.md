# Qwen Local AI Moqui XML Generation Toolkit

**Date:** March 7, 2026

## Overview
Today, we developed a powerful, self-hosted AI integration using the local `qwen2.5-coder:7b` model running on Ollama. The objective of this toolkit is to automatically interpret human-readable specifications written in Markdown ("Blueprints") and legacy Moqui XML data, and natively generate Moqui 4.0 artifact files, heavily accelerating the new "aitree" component development.

We introduced two core Python scripts, complementary shell scripts, and agent workflows to orchestrate these transformations completely locally without external API dependencies.

## Core Capabilities Developed

### 1. Feature Artifact Generation (`blueprint-gen.py`)
This script acts as a compiler that translates human-authored Markdown UI Blueprints into fully valid Moqui XML screen definitions or JavaScript files.
- **Context Injection:** The python script dynamically gathers context from the project structure before prompting Qwen. It injects the `moqui-ai-screen.xsd` schema, the available Quasar/Moqui Vue UI macro templates (`MoquiAiScreenMacros.qvt.ftl`), the `AiTreeEntities.xml` framework definitions, and even recursive sibling blueprints (to understand parent/child tab structures).
- **Golden Samples:** A predefined "Golden Sample" structure mapping a mock "Staff Meeting" Blueprint to a fully qualified `<screen>` XML is sent in the system prompt to guide the model's output syntax and teach it how to use `<subscreens-active/>` and custom Vue macros.
- **Workflow Command (`/blueprint-gen`):** The process was wrapped into an agent workflow located at `.agent/workflows/blueprint-gen.md` allowing us to instantly call `/blueprint-gen` against any target `.md` requirement file to magically materialize the matching `.xml` Moqui screen alongside it.

### 2. Legacy Data Migration (`qwen-data-transform.py`)
This script tackles the complex task of migrating old legacy data structures from a previous `studdle` implementation into the new structured `aitree` format.
- **Inputs:** It reads a predefined block of legacy XML (`wepop.xml`), a seed file (`agendaSeedData.xml`), and the new target entity definitions (`AiTreeEntities.xml`).
- **Semantic Mapping:** Instead of writing dozens of fragile regexes or XSLT mappings, the script asks Qwen to semantically understand that a legacy `<sh.AgendaTopic agendaTopicId="X">` represents a new `<aitree.meeting.AgendaMessage agendaMessageId="X">`, and an attached `<sh.SHContent title="Y">` becomes the new `<aitree.meeting.AgendaMessageContent title="Y">`. 
- **Deterministic Wrappers:** It statically wraps the output generated by Qwen in the necessary `<entity-facade-xml>` root tags and dynamically inserts mandatory foundational infrastructural records (like `BASE_REPO` AgendaContainers) before saving it as `agendaContainerData.xml`.
- **Workflow Command (`/data-load`):** This data streaming execution was also packaged as an accessible workflow `.agent/workflows/data-load.md` integrated with `bin/data-load.sh`.

## Conclusion
The introduction of these tools effectively transforms the Moqui software development life cycle. Rather than directly coding repetitive XML structures, developers can now write conceptual UI/UX behaviors in simple markdown blueprints or provide semantic mapping rules, and securely rely on the local Qwen model to confidently generate Moqui 4.0 compliant, syntactically-perfect XML artifacts and data files!
