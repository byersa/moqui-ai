## CRITICAL: Test Execution Command

**ALWAYS run tests with `cleanDb` and `load` commands from the root project directory.**

**Resolve placeholders**: Read `overlay-*-project-profile.md` in `runtime/component/moqui-agent-os/` for concrete data type names and per-component test commands.

```bash
# Standard pattern - always use this format for ANY component
./gradlew cleanDb load runtime:component:{component-name}:test -Ptypes=seed,seed-initial,{l10n}-install,{project}-test,{project}-demo,{component}-test,{component-name}-test
```

**Note:** The `{component-name}-test` type is always included by convention. If it doesn't exist for a component, it is safely ignored without error.

**Why This is Critical:**
- `cleanDb` ensures a clean database state for each test run
- `load` with `-Ptypes=` loads the specific seed and test data needed
- Running tests without this pattern causes unpredictable failures from stale data
- **NEVER** run just `./gradlew :runtime:component:{name}:test` without cleanDb and load

📄 **Full Reference**: See `testing-guide.md` for complete data type documentation.

## Common Testing Gotchas

### Auto-Generated Create Services Don't Return User-Provided PKs

**IMPORTANT**: When using auto-generated `create#Entity` services for entities with user-provided composite primary keys (not auto-generated `id-long` types), the service returns an **empty map** `[:]`. It does NOT return the PK fields back because they weren't generated by the service.

```groovy
// WRONG - This assertion will ALWAYS fail
def result = ec.service.sync().name("create#mycompany.myapp.FieldConfiguration")
    .parameters([partyId: "ORG1", fieldDefinitionId: "FIELD1", isRequired: "Y"])
    .call()
result.partyId == "ORG1"  // FAILS! result is [:] (empty map)

// CORRECT - Verify no errors, then query the entity to confirm creation
def result = ec.service.sync().name("create#mycompany.myapp.FieldConfiguration")
    .parameters([partyId: "ORG1", fieldDefinitionId: "FIELD1", isRequired: "Y"])
    .call()
!ec.message.hasError()  // First verify no service errors

// Then verify entity was actually created
def created = ec.entity.find("mycompany.myapp.FieldConfiguration")
    .condition("partyId", "ORG1")
    .condition("fieldDefinitionId", "FIELD1")
    .one()
created != null
created.isRequired == "Y"
```

**When PKs ARE returned**: Auto-generated create services DO return PK values when the entity uses auto-generated primary keys like `id-long` type. In that case, `result.entityId` will contain the generated ID.

### View Entity Joins Can Cause Empty Results

Non-optional view entity joins (without `join-optional="true"`) require ALL joined entities to exist. If any required join fails, the entire row is excluded from results.

**Common symptom**: Service returns empty/null when you expect data.

```xml
<!-- This join REQUIRES PartyIdentification to exist -->
<member-entity entity-alias="VENDORID" entity-name="mantle.party.PartyIdentification"
               join-from-alias="ORDENCOMPRA">
    <key-map field-name="vendorPartyId" related="partyId"/>
    <entity-condition>
        <econdition field-name="partyIdTypeEnumId" operator="in"
                    value="PtidNationalTaxId,PtidUniqueNationalId,PtidPassport"/>
    </entity-condition>
</member-entity>
```

**Fix**: Ensure test setup creates ALL entities required by view entity joins:

```groovy
// In test setup - create the required PartyIdentification record
ec.service.sync().name("create#mantle.party.PartyIdentification")
    .parameters([
        partyId: testVendorPartyId,
        partyIdTypeEnumId: "PtidNationalTaxId",  // Must match view entity condition
        idValue: "76.000.000-0"
    ])
    .call()
```

### Timestamp Ordering is Unreliable Within Same Transaction

When ordering by `lastUpdatedStamp`, `createdDate`, or similar timestamp fields, records created within the same transaction may have identical or unpredictable timestamps. `Thread.sleep()` does NOT guarantee different timestamps within a single database transaction.

**Common symptom**: Tests expecting "most recent" record get wrong results intermittently.

```groovy
// UNRELIABLE - All records may have same timestamp
def record1 = createRecord()
Thread.sleep(10)  // Does NOT help within same transaction
def record2 = createRecord()
Thread.sleep(10)
def record3 = createRecord()

// Query expecting record3 as "newest" may return any of them
def newest = ec.entity.find("MyEntity")
    .orderBy("-lastUpdatedStamp")
    .one()  // May NOT be record3!
```

**Solutions**:

1. **Simplify tests**: Don't rely on "most recent" ordering when creating multiple records
   ```groovy
   // Use only 2 records - one previous, one current
   def previousRecord = createRecord()
   Thread.sleep(100)
   def currentRecord = createRecord()
   // Service should find previousRecord (the only other one)
   ```

2. **Use explicit ordering field**: Add a sequence/order field you control
   ```groovy
   def record1 = createRecord([sequenceNum: 1])
   def record2 = createRecord([sequenceNum: 2])
   // Order by sequenceNum instead of timestamp
   ```

3. **Use separate transactions** (more complex):
   ```groovy
   ec.transaction.commit()
   def record1 = createRecord()
   ec.transaction.commit()
   Thread.sleep(100)
   ec.transaction.begin(null)
   def record2 = createRecord()
   // Now timestamps will be different
   ```

## Service Testing Patterns

### Basic Service Test
```groovy
def "service creates valid entity"() {
    when: "calling create service"
    def result = sf.sync().name("example.create#Entity")
            .parameter("name", "Test Entity")
            .parameter("value", 100)
            .call()

    then: "service succeeds"
    !sf.isError(result)
    result.entityId != null

    and: "entity is created correctly"
    def entity = ec.entity.find("Entity")
            .condition("entityId", result.entityId)
            .one()
    entity != null
    entity.name == "Test Entity"
    entity.value == 100
}
```

### Parameter Validation Testing
```groovy
def "service validates input parameters correctly"() {
    expect: "parameter validation"
    def result = sf.sync().name("example.validate#Input")
            .parameters(inputParams)
            .call()

    isError == sf.isError(result)
    if (isError) {
        result.errors.any { it.contains(expectedError) }
    }

    where:
    inputParams                              | isError | expectedError
    [requiredField: "valid"]                 | false   | null
    [:]                                      | true    | "requiredField"
    [requiredField: ""]                      | true    | "requiredField"
    [requiredField: "valid", numField: -1]   | true    | "positive"
}
```

### State Transition Testing
```groovy
def "order state transitions work correctly"() {
    given: "order in initial state"
    def order = createTestOrder([statusId: "OrderPlaced"])

    when: "transitioning to next valid state"
    def result = sf.sync().name("order.process#Order")
            .parameter("orderId", order.orderId)
            .call()

    then: "transition succeeds"
    !sf.isError(result)
    def updatedOrder = ec.entity.refresh(order)
    updatedOrder.statusId == "OrderProcessing"

    when: "attempting invalid transition"
    def invalidResult = sf.sync().name("order.ship#Order")
            .parameter("orderId", order.orderId)
            .call()

    then: "invalid transition is rejected"
    sf.isError(invalidResult)
    invalidResult.errors.any { it.contains("Invalid status transition") }
}
```

def "entity relationships are properly configured"() {
    given: "related entities"
    def parent = ec.entity.makeValue("ParentEntity")
            .set("name", "Test Parent")
            .sequencedIdPrimary()
            .create()

    def child = ec.entity.makeValue("ChildEntity")

    then: "relationships work correctly"

    parentFromChild.parentId == parent.parentId
    childrenFromParent.size() == 1
}

## Integration Testing Patterns

### Complete Workflow Test
```groovy
def "complete business workflow executes correctly"() {
    given: "workflow prerequisites"
    def customer = createTestCustomer()
    def product = createTestProduct([quantityOnHand: 10])

    when: "executing complete workflow"
    def orderResult = sf.sync().name("order.create#Order")
            .parameter("customerId", customer.customerId)
            .call()

    def itemResult = sf.sync().name("order.add#OrderItem")
            .parameter("orderId", orderResult.orderId)
            .parameter("productId", product.productId)
            .parameter("quantity", 2)
            .call()

    def fulfillResult = sf.sync().name("order.fulfill#Order")
            .parameter("orderId", orderResult.orderId)
            .call()

    then: "all steps complete successfully"
    [orderResult, itemResult, fulfillResult].every { !sf.isError(it) }

    and: "verify final state"
    def finalOrder = ec.entity.find("OrderHeader")
            .condition("orderId", orderResult.orderId)
            .one()
    finalOrder.statusId == "OrderFulfilled"

    def updatedProduct = ec.entity.refresh(product)
    updatedProduct.quantityOnHand == 8
}
```

## Test Utilities

### Common Assertion Helpers
```groovy
class TestPatterns {
    static def assertServiceSuccess(ServiceFacade sf, Map result, String operation = "operation") {
        assert !sf.isError(result), "${operation} should succeed but returned: ${result.errors}"
    }

    static def assertServiceError(ServiceFacade sf, Map result, String expectedError) {
        assert sf.isError(result), "Operation should fail but succeeded"
        assert result.errors.any { it.contains(expectedError) },
               "Expected error containing '${expectedError}' but got: ${result.errors}"
    }

    static def assertEntityExists(ExecutionContext ec, String entityName, Map conditions) {
        def entity = ec.entity.find(entityName)
        conditions.each { field, value ->
            entity.condition(field, value)
        }
        def found = entity.one()
        assert found != null, "Expected ${entityName} to exist"
        return found
    }
}
```

### Test Data Builder Pattern
```groovy
class TestDataBuilder {
    private ExecutionContext ec
    private Map data = [:]

    TestDataBuilder(ExecutionContext ec) {
        this.ec = ec
    }

    static def customer(ExecutionContext ec) {
        return new TestDataBuilder(ec).withDefaults([
            firstName: "Test",
            lastName: "Customer"
        ])
    }

    def withDefaults(Map defaults) {
        data.putAll(defaults)
        return this
    }

    def with(String field, Object value) {
        data[field] = value
        return this
    }

    def build() {
        return ec.entity.makeValue("Customer")
                .setAll(data)
                .sequencedIdPrimary()
                .create()
    }
}

// Usage:
def customer = TestDataBuilder.customer(ec)
        .with("firstName", "John")
        .build()
```

## DTE Expected Value Calculation

When testing Chilean DTE generation services, expected values must account for all 4 input/output mode combinations:

### Input/Output Mode Matrix

| Input Mode | Output Mode | MontoItem Calculation | MntTotal Calculation |
|------------|-------------|----------------------|---------------------|
| Gross (`preciosIncluyenIva=true`) | Gross XML | `round(qty × grossPrice)` | Sum of MontoItems |
| Net (`preciosIncluyenIva=false`) | Gross XML | `round(qty × netPrice × 1.19)` | Sum of MontoItems |
| Gross (`preciosIncluyenIva=true`) | Net XML | `round(qty × grossPrice / 1.19)` | MntNeto + IVA |
| Net (`preciosIncluyenIva=false`) | Net XML | `round(qty × netPrice)` | MntNeto + IVA |

### Expected Value Calculation Pattern

```groovy
BigDecimal IVA_FACTOR = new BigDecimal("1.19")
BigDecimal IVA_RATE = new BigDecimal("0.19")

BigDecimal expectedTotal
BigDecimal expectedMntNeto

if (!generarXmlMontosNetos) {
    // Gross XML mode: MontoItems are gross amounts
    if (preciosIncluyenIva) {
        // Gross input: amount is already gross
        BigDecimal montoItem = (qty * inputAmount).setScale(0, RoundingMode.HALF_UP)
        expectedTotal = montoItem
        expectedMntNeto = (montoItem / IVA_FACTOR).setScale(0, RoundingMode.HALF_UP)
    } else {
        // Net input: convert to gross with intermediate rounding
        BigDecimal grossPrice = (inputAmount * IVA_FACTOR).setScale(6, RoundingMode.HALF_UP)
        BigDecimal montoItem = (qty * grossPrice).setScale(0, RoundingMode.HALF_UP)
        expectedTotal = montoItem
        expectedMntNeto = (qty * inputAmount).setScale(0, RoundingMode.HALF_UP)
    }
} else {
    // Net XML mode: MontoItems are net amounts, MntTotal = MntNeto + IVA
    if (preciosIncluyenIva) {
        // Gross input: convert to net
        BigDecimal netPrice = inputAmount.divide(IVA_FACTOR, 6, RoundingMode.HALF_UP)
        expectedMntNeto = (qty * netPrice).setScale(0, RoundingMode.HALF_UP)
    } else {
        // Net input: amount is already net
        expectedMntNeto = (qty * inputAmount).setScale(0, RoundingMode.HALF_UP)
    }
    BigDecimal expectedIva = (expectedMntNeto * IVA_RATE).setScale(0, RoundingMode.HALF_UP)
    expectedTotal = expectedMntNeto + expectedIva
}
```

### Common Testing Pitfalls

1. **Using same expected calculation for all modes**: Each combination requires different logic
2. **Forgetting intermediate rounding**: Net-to-gross conversion needs `.setScale(6, HALF_UP)` before MontoItem calculation
3. **Assuming MntTotal = MntNeto + IVA**: Only true for net XML mode; gross XML uses sum of MontoItems

**Reference**: See your project's component-level `.agent-os/` directory for domain-specific patterns documentation.

## Available Demo Party IDs for Foreign Keys

When writing tests or demo data that reference `partyId` (a foreign key to `mantle.party.Party`), you **must** use party IDs that already exist in the loaded seed/demo data. Using non-existent IDs causes referential integrity constraint violations.

### Valid partyId Values (from mantle-usl ZbaOrganizationDemoData.xml)

| partyId | Type | Description |
|---------|------|-------------|
| `ORG_ZIZI_CORP` | Organization | Ziziwork Industries (main org) |
| `ORG_ZIZI_SALES` | Organization | Ziziwork Sales |
| `ORG_ZIZI_HR` | Organization | Ziziwork Human Resources |
| `ORG_ZIZI_SERVICES` | Organization | Ziziwork Services |
| `ORG_ZIZI_RETAIL` | Organization | Ziziwork Retail |
| `ORG_ZIZI_DEVA` | Organization | Ziziwork Dev Team A |
| `ORG_ZIZI_DEVB` | Organization | Ziziwork Dev Team B |
| `ORG_ZIZI_JD` | Person | Joe Developer (Ziziwork employee) |
| `ORG_ZIZI_BD` | Person | Bob Developer (Ziziwork employee) |
| `ORG_ACME` | Organization | Another Company Making Everything |
| `ORG_ACME_BA` | Person | Bob Acme |
| `ORG_ACME_SA` | Person | Sheldon Acme |
| `ORG_BLUTH` | Organization | Bluth Corp |
| `ORG_BLUTH_GOB` | Person | Gob Bluth |
| `CustJqp` | Person | Customer John Q. Public |
| `EX_JOHN_DOE` | Person | John Doe |
| `ZiddlemanInc` | Organization | Ziddleman & Sons Suppliers Inc. |
| `JoeDist` | Organization | Joe Distributors |

### Carriers (from mantle-udm CarrierData.xml)

| partyId | Description |
|---------|-------------|
| `UPS` | United Parcel Service |
| `FedEx` | FedEx Corporation |
| `DHLX` | DHL Express |
| `USPS` | United States Postal Service |

### Recommended Usage in Tests & Demo Data

- **For employee/person records**: Use `ORG_ZIZI_JD`, `ORG_ZIZI_BD`, `EX_JOHN_DOE`, `CustJqp`
- **For vendor/customer org records**: Use `ORG_ZIZI_RETAIL`, `ORG_ACME`, `ZiddlemanInc`
- **For internal org records**: Use `ORG_ZIZI_CORP`, `ORG_ZIZI_HR`
- **NEVER invent partyId values** -- always reference IDs from the tables above
