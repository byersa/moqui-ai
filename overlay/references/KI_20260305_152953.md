# Configuring a Local Qwen/Ollama PC as a Project Memory Engine

## Context
The development team is setting up a local environment for a Qwen or Ollama project, which requires a memory engine to store and retrieve data efficiently. The goal is to ensure that the setup is portable and persistent across different developers' machines.

## Problem
Developers faced issues with sudo password prompts in background processes when starting/stopping Ollama and loading/unloading models. This required manual intervention and was inconvenient for automation.

## Solution (Specific Moqui XML/Script patterns)
1. **Bash Scripts for Background Processes**:
    - Created bash scripts to automate the starting and stopping of Ollama and model loading/unloading.
    - Used the `nohup` command to run scripts in the background, which avoids sudo password prompts.
    - Example of a bash script to start Ollama:
      ```bash
      #!/bin/bash
      nohup ollama start &
      ```
    - Example of a bash script to stop Ollama:
      ```bash
      #!/bin/bash
      pkill ollama
      ```

2. **Systemd Service for Persistent Startup**:
    - Created a systemd service to manage the Ollama service as a system-level process.
    - Configured the service to start on boot and to handle authentication via GUI popups instead of sudo.
    - Example systemd service file (`/etc/systemd/system/ollama.service`):
      ```ini
      [Unit]
      Description=Ollama Service
      After=network.target

      [Service]
      Type=simple
      ExecStart=/usr/local/bin/ollama start
      ExecStop=/usr/local/bin/ollama stop
      Restart=on-failure
      User=youruser
      Environment=DISPLAY=:0
      Environment=SSH_AUTH_SOCK=/tmp/ssh-$(whoami)-agent-$(date +%s).sock

      [Install]
      WantedBy=multi-user.target
      ```

3. **Root-Level Shortcuts for Setup**:
    - Established a `/setup` workflow to anchor these scripts and systemd service as root-level shortcuts for new users.
    - Ensured that the agent and environment are portable and persistent across developers by configuring the scripts and service to run as the current user.
    - Example of a bash script to setup the Ollama service:
      ```bash
      #!/bin/bash
      cp /path/to/ollama.service /etc/systemd/system/
      systemctl daemon-reload
      systemctl enable ollama
      systemctl start ollama
      ```

## Prevention (How to avoid re-doing this work)
1. **Standardize Configuration Files**:
    - Create standardized configuration files for Ollama and the bash scripts.
    - Ensure that all developers follow the same template for their bash scripts and systemd service files.

2. **Automate Setup with Docker**:
    - Consider using Docker to containerize the Ollama environment.
    - This will ensure that the setup is consistent across all developers' machines and avoids the need for manual configuration.

3. **Document and Share Best Practices**:
    - Document the setup process and share best practices with the team.
    - Provide a checklist of steps for setting up the Ollama environment and ensure that everyone follows it.

By following these steps, the development team can ensure that the Ollama/Qwen project environment is portable, persistent, and easy to configure for new developers.
